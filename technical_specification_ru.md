## 1. Общие сведения

### 1.1. Назначение документа

Настоящее техническое задание описывает полный перечень работ по созданию серверной части (бэкенда), базы данных, подключению пользовательского интерфейса (фронтенда) к серверу и развёртыванию системы управления проектами в рабочей среде.

Документ предназначен для согласования объёма, последовательности и критериев качества работ между заказчиком и командой разработки.

### 1.2. Описание продукта

**Система управления проектами** — это веб-приложение для архитектурных и дизайнерских бюро, которое позволяет:

- Вести реестр проектов с бюджетами, сроками и командами
- Управлять задачами и подзадачами с приоритетами и статусами
- Вести базу клиентов и контактных лиц
- Отслеживать рабочее время сотрудников
- Анализировать продуктивность команды через дашборды и графики
- Работать с файлами и документами по каждому проекту
- Управлять командой: роли, навыки, статусы сотрудников

### 1.3. Термины и определения

| Термин | Значение |
|--------|----------|
| **Фронтенд** | Пользовательский интерфейс — то, что видит и с чем взаимодействует пользователь в браузере |
| **Бэкенд (сервер)** | Серверная часть — программа, которая обрабатывает запросы, хранит данные и обеспечивает безопасность |
| **База данных** | Централизованное хранилище всей информации системы (проекты, задачи, пользователи и т.д.) |
| **API** | Программный интерфейс — набор команд, через которые интерфейс общается с сервером |
| **JWT** | Метод безопасной авторизации: система выдаёт пользователю цифровой «пропуск», который подтверждает его личность |
| **Развёртывание** | Размещение готового приложения на рабочем сервере, доступном пользователям |
| **Миграция** | Перенос данных из старого формата (тестовые данные) в новую базу данных |

## 2. Текущее состояние продукта

### 2.1. Что уже сделано

Полностью разработан пользовательский интерфейс приложения, который включает:

| Раздел | Что реализовано |
|--------|----------------|
| **Панель проектов** | Таблица проектов с фильтрацией, сортировкой, детальная карточка проекта, управление бюджетом |
| **Задачи** | Таблица задач с подзадачами, редактирование прямо в таблице, чек-листы, зависимости между задачами |
| **Команда** | Дашборд команды со статусами сотрудников (работает/перерыв/оффлайн), профиль сотрудника, управление командой |
| **Клиенты** | Список клиентов с группами, метками, избранным, контактные лица, детальная карточка |
| **Аналитика** | Графики продуктивности, распределение задач по статусам и приоритетам, учёт времени |
| **Настройки** | 5 визуальных тем оформления, 3 языка (русский, английский, узбекский), пользовательские настройки |

### 2.2. Что отсутствует

На данный момент всё приложение работает на **демонстрационных данных** — информация хранится только в памяти браузера и пропадает при обновлении страницы. Отсутствуют:

- Серверная часть для обработки и хранения данных
- База данных для постоянного хранения информации
- Система авторизации (вход/регистрация пользователей)
- Возможность совместной работы нескольких пользователей
- Загрузка и хранение файлов
- Уведомления
- Обновление данных в реальном времени (когда один пользователь вносит изменение, другие видят его мгновенно)

## 3. Цели и задачи

### 3.1. Бизнес-цели

1. **Превратить прототип в полноценный рабочий продукт**, пригодный для ежедневного использования командой
2. **Обеспечить сохранность данных** — вся информация хранится на сервере и не теряется
3. **Дать возможность совместной работы** — несколько сотрудников одновременно работают в системе
4. **Разграничить права доступа** — руководители, менеджеры и сотрудники видят и могут делать разные вещи
5. **Подготовить продукт к масштабированию** — при росте компании система работает стабильно

### 3.2. Технические задачи

1. Спроектировать и развернуть базу данных для хранения всех сущностей системы
2. Разработать серверную часть с полным набором операций для каждого раздела
3. Реализовать систему авторизации и разграничения прав
4. Подключить интерфейс к серверу вместо демонстрационных данных
5. Реализовать обновления в реальном времени
6. Настроить загрузку и хранение файлов
7. Развернуть систему на рабочем сервере

| Этап | Название | Суть |
|------|----------|------|
| 1 | Серверная инфраструктура | Настройка сервера, базы данных, системы входа/регистрации |

| 3 | Связи и файлы | Участники задач, зависимости, чек-листы, комментарии, загрузка файлов |
| 4 | Учёт времени | Фиксация рабочих часов, таймер, отчёты по времени |
| 5 | Реальное время и аналитика | Мгновенные обновления, уведомления, поиск, экспорт отчётов |

| 7 | Развёртывание | Размещение на сервере, настройка безопасности, запуск |

## 5. Этап 1 — Серверная инфраструктура и база данных

### 5.1. Что будет сделано

Создание фундамента системы: серверная программа, база данных, система входа и регистрации пользователей.

### 5.2. Состав работ

#### 5.2.1. Настройка серверного приложения

- Создание проекта серверной части на языке TypeScript (Node.js)
- Настройка структуры папок и модулей
- Подключение к базе данных через систему управления схемами (ORM)
- Настройка правил безопасности для взаимодействия интерфейса с сервером (CORS)

#### 5.2.2. Создание базы данных

В базе данных будет создано **15 таблиц** для хранения всей информации системы:

| Таблица | Назначение | Что хранит |
|---------|-----------|------------|
| **Организации** | Компании-пользователи системы | Название, логотип, лимит бюджета, настройки |
| **Пользователи** | Сотрудники компании | Имя, email, роль, должность, отдел, навыки, рейтинг, фото |
| **Клиенты** | Заказчики проектов | Название, тип, контакты, адрес, ИНН, группа, метки |
| **Контактные лица** | Представители клиентов | Имя, email, телефон, должность, к какому клиенту относится |
| **Проекты** | Архитектурные/дизайнерские проекты | Название, клиент, даты, статус, тип, площадь, бюджет, прогресс |
| **Участники проектов** | Кто работает на проекте | Связка пользователь-проект, роль на проекте |
| **Задачи** | Работы по проектам | Название, проект, исполнитель, статус, приоритет, тип работы, сроки |
| **Участники задач** | Дополнительные исполнители | Связка пользователь-задача |
| **Зависимости задач** | Какая задача блокирует какую | Связки «блокирует — заблокирована» |
| **Чек-листы** | Пункты выполнения внутри задачи | Текст пункта, выполнен/не выполнен, порядок |
| **Учёт времени** | Записи о рабочих часах | Кто, на какую задачу, дата, часы, оплачиваемо или нет |
| **Комментарии** | Обсуждение задач | Автор, задача, текст, дата |
| **Журнал действий** | История всех изменений | Кто, что, когда изменил и какие были старые/новые значения |
| **Файлы** | Вложения к проектам/задачам/клиентам | Имя файла, размер, тип, кто загрузил |
| **Уведомления** | Оповещения для пользователей | Заголовок, текст, тип, прочитано/нет |

#### 5.2.3. Система авторизации

- **Регистрация** — создание новой организации и первого администратора
- **Вход в систему** — по email и паролю; система выдаёт защищённый цифровой «пропуск» (токен)
- **Автоматическое продление сессии** — пользователю не нужно входить заново каждые 15 минут
- **Выход** — безопасное завершение сессии
- **Смена пароля** — с подтверждением текущего пароля
- **Профиль** — получение данных о текущем пользователе

#### 5.2.4. Наполнение тестовыми данными

Перенос всех текущих демонстрационных данных из интерфейса в базу данных для проверки работоспособности:

- 1 организация
- 8–16 пользователей с разными ролями
- 8 проектов с различными статусами
- 10+ задач с подзадачами
- 13 клиентов

### 5.3. Результат этапа

- Работающий сервер, принимающий запросы
- База данных со всеми таблицами и тестовыми данными
- Возможность зарегистрироваться, войти в систему и получить свой профиль
- Документация по API (описание всех доступных команд сервера)

## 6. Этап 2 — Основные операции с данными

### 6.1. Что будет сделано

Реализация всех базовых операций для четырёх ключевых разделов: пользователи, проекты, задачи и клиенты. После этого этапа через сервер можно будет создавать, просматривать, редактировать и удалять все основные объекты системы.

### 6.2. Состав работ

#### 6.2.1. Управление пользователями (сотрудниками)

| Операция | Описание | Кто может |
|----------|----------|-----------|
| Список сотрудников | С фильтрацией по статусу, отделу и поиском по имени | Все |
| Профиль сотрудника | Полная информация + статистика: задачи, часы, проекты | Все |
| Создание сотрудника | Приглашение нового пользователя в систему | Администратор, менеджер |
| Редактирование профиля | Изменение данных сотрудника | Сам пользователь или администратор |
| Деактивация | Мягкое удаление — пользователь не удаляется, но не может войти | Администратор |
| Статус онлайн | Обновление статуса: работает / перерыв / оффлайн | Сам пользователь |
| Загрузка фото | Установка аватара профиля | Сам пользователь |
| Настройки | Сохранение темы, языка, конфигурации колонок таблиц | Сам пользователь |

#### 6.2.2. Управление проектами

| Операция | Описание | Кто может |
|----------|----------|-----------|
| Список проектов | С фильтрацией по статусу, клиенту, поиском; скрытие архивных | Все |
| Детали проекта | Полная карточка с командой, количеством задач, бюджетом | Все |
| Создание проекта | Название, клиент, даты, тип, площадь, бюджет | Администратор, менеджер |
| Редактирование | Обновление любого поля проекта | Администратор, менеджер |
| Архивирование | Проект скрывается из активного списка, но не удаляется | Администратор, менеджер |
| Управление командой | Добавление/удаление участников проекта, назначение ролей | Администратор, менеджер |
| Бюджет проекта | Просмотр и обновление бюджета с проверкой лимита организации | Администратор, менеджер |
| Закрепление | Проект отображается в верхней части боковой панели | Все |
| История изменений | Журнал всех действий по проекту | Все |

#### 6.2.3. Управление задачами

| Операция | Описание | Кто может |
|----------|----------|-----------|
| Список задач | Плоский вид или дерево; фильтры по проекту, статусу, исполнителю, приоритету, типу работ | Все |
| Детали задачи | Полная карточка с подзадачами, зависимостями, чек-листом | Все |
| Создание задачи | Проект, название, исполнитель, статус, приоритет, тип работ, сроки | Все (свои или назначенные) |
| Редактирование | Обновление любого поля — поддержка редактирования прямо в таблице | Все (свои или назначенные) |
| Удаление | Удаление задачи вместе со всеми подзадачами | Администратор, менеджер |
| Дублирование | Создание копии задачи с подзадачами | Все |
| Быстрое обновление статуса | Смена статуса в один клик | Все (свои или назначенные) |
| Приёмка | Утверждение / отклонение / отправка на доработку | Администратор, менеджер |
| Прогресс | Обновление процента выполнения | Все (свои или назначенные) |
| Переназначение | Смена исполнителя | Администратор, менеджер |
| Перетаскивание | Изменение порядка и вложенности задач | Все |
| Подзадачи | Создание задач внутри задач (неограниченная вложенность) | Все |

**Статусы задач:** Старт → В работе → Горит → Завершена → Просрочена → Отменена

**Типы работ:** Архитектура, Дизайн интерьера, Дизайн экстерьера, Ландшафт, Рабочие чертежи, 3D-визуализация, Авторский надзор, Документация, Инженерия, Координация с клиентом

#### 6.2.4. Управление клиентами

| Операция | Описание | Кто может |
|----------|----------|-----------|
| Список клиентов | С поиском, фильтрацией по группе, наличию проектов | Администратор, менеджер |
| Детали клиента | Полная карточка с контактными лицами и количеством проектов | Администратор, менеджер |
| Создание клиента | Название, тип, контакты, адрес, ИНН, группа, метки | Администратор, менеджер |
| Редактирование | Обновление данных клиента | Администратор, менеджер |
| Удаление | Удаление клиента | Администратор, менеджер |
| Проекты клиента | Список всех проектов данного клиента | Все |
| Избранное | Пометка клиента как избранного | Администратор, менеджер |

#### 6.2.5. Управление бюджетом организации

| Операция | Описание | Кто может |
|----------|----------|-----------|
| Обзор бюджета | Общий лимит, потрачено, остаток, разбивка по проектам | Все |
| Изменение лимита | Установка общего бюджетного лимита организации | Администратор |

### 6.3. Результат этапа

- Полностью рабочие операции создания, чтения, обновления и удаления для пользователей, проектов, задач и клиентов
- Управление бюджетом с проверкой лимитов
- Все фильтры и поиск работают на стороне сервера

## 7. Этап 3 — Связи между объектами и работа с файлами

### 7.1. Что будет сделано

Реализация дополнительных сущностей, связывающих основные объекты, и подключение хранилища файлов.

### 7.2. Состав работ

#### 7.2.1. Участники задач

- Добавление дополнительных исполнителей к задаче (помимо основного)
- Просмотр списка участников
- Удаление участника из задачи

#### 7.2.2. Зависимости между задачами

- Установка связи «задача A блокирует задачу B»
- Просмотр, какие задачи блокируют текущую и какие она блокирует
- Удаление зависимости
- Защита от циклических зависимостей (A блокирует B, B блокирует A — запрещено)

#### 7.2.3. Чек-листы внутри задач

- Создание пунктов чек-листа
- Отметка пункта как выполненного / невыполненного
- Переименование пункта
- Удаление пункта
- Изменение порядка пунктов перетаскиванием

#### 7.2.4. Комментарии к задачам

- Добавление комментария
- Редактирование своего комментария
- Удаление комментария
- Постраничная загрузка (для задач с большим количеством обсуждений)

#### 7.2.5. Контактные лица клиентов

- Добавление контактного лица к клиенту
- Редактирование контактных данных
- Удаление контактного лица
- Отметка основного контактного лица

#### 7.2.6. Загрузка и хранение файлов

- Загрузка файлов к проектам, задачам и клиентам
- Просмотр списка файлов по каждому объекту
- Скачивание файлов через защищённые ссылки
- Удаление файлов
- Хранение файлов в облачном хранилище (S3-совместимое)
- Структурированное хранение: файлы организованы по папкам организации, проектов и задач

### 7.3. Результат этапа

- Все связи между объектами работают через сервер
- Файлы загружаются, хранятся и доступны для скачивания
- Комментарии и чек-листы полностью функциональны

## 8. Этап 4 — Учёт рабочего времени

### 8.1. Что будет сделано

Система фиксации рабочего времени с возможностью ручного ввода и таймера, а также агрегированные отчёты по времени.

### 8.2. Состав работ

#### 8.2.1. Записи рабочего времени

| Операция | Описание |
|----------|----------|
| Список записей | С фильтрацией по пользователю, проекту, задаче, периоду |
| Создание записи | Задача, проект, дата, количество часов, описание, оплачиваемо или нет |
| Редактирование | Изменение существующей записи |
| Удаление | Удаление записи |
| Запуск таймера | Начало отсчёта времени по конкретной задаче |
| Остановка таймера | Завершение отсчёта, автоматический расчёт часов |

#### 8.2.2. Отчёты по времени

| Отчёт | Описание |
|-------|----------|
| Сводка по проектам | Сколько часов потрачено на каждый проект за период |
| Сводка по сотрудникам | Сколько часов отработал каждый сотрудник |
| Сводка по типам работ | Распределение времени между архитектурой, дизайном, визуализацией и т.д. |

### 8.3. Результат этапа

- Полноценный учёт рабочего времени с таймером
- Агрегированные отчёты по разным разрезам
- Данные аналитического дашборда формируются из реальных записей

## 9. Этап 5 — Обновления в реальном времени и аналитика

### 9.1. Что будет сделано

Мгновенная доставка изменений всем подключённым пользователям, система уведомлений, глобальный поиск и расширенная аналитика.

### 9.2. Состав работ

#### 9.2.1. Обновления в реальном времени

Когда один пользователь вносит изменение, все остальные видят его мгновенно без перезагрузки страницы:

| Событие | Что видят другие пользователи |
|---------|-------------------------------|
| Сотрудник стал «онлайн» | На дашборде команды его статус меняется на «работает» |
| Задача обновлена | В таблице задач меняется статус / прогресс / исполнитель |
| Новая задача создана | Задача появляется в списке у всех участников проекта |
| Задача удалена | Задача исчезает из списка |
| Проект обновлён | Изменения отражаются в таблице проектов и боковой панели |
| Бюджет изменён | Обновляется виджет бюджета |
| Зафиксировано время | Обновляются часы на дашборде команды |

#### 9.2.2. Система уведомлений

- Получение уведомлений о назначении на задачу, приближении дедлайна, новых комментариях
- Просмотр списка уведомлений (прочитанные / непрочитанные)
- Отметка одного или всех уведомлений как прочитанных
- Мгновенная доставка уведомления через реальное время

#### 9.2.3. Глобальный поиск

- Поиск по всем разделам одновременно: проекты, задачи, клиенты, пользователи
- Фильтрация результатов по типу объекта
- Быстрый поиск по ключевому слову

#### 9.2.4. Расширенная аналитика

| Отчёт | Описание |
|-------|----------|
| Обзор | Завершённые задачи, активные, просроченные, общие часы, среднее время на задачу |
| Динамика завершения задач | График тренда по дням / неделям |
| Распределение задач | Сколько задач в каждом статусе и приоритете |
| Продуктивность команды | Завершённые задачи, часы и эффективность каждого сотрудника |
| Время по проектам | Топ проектов по затраченному времени |
| Время по типам работ | Сколько часов уходит на архитектуру, дизайн, визуализацию и т.д. |
| Недельная продуктивность | Задачи и часы по дням недели |
| Ежемесячный отчёт | Разбивка по проектам с часами, задачами, участниками |
| Экспорт | Выгрузка отчётов в формате CSV или Excel |

### 9.3. Результат этапа

- Все пользователи видят изменения мгновенно
- Работающая система уведомлений
- Полноценный поиск по всей системе
- Аналитические дашборды формируются из реальных данных с возможностью экспорта

## 10. Этап 6 — Интеграция интерфейса с сервером

### 10.1. Что будет сделано

Это самый масштабный этап: полная замена демонстрационных данных на реальные данные с сервера. Каждый экран интерфейса будет подключён к серверу.

### 10.2. Состав работ

#### 10.2.1. Создание слоя взаимодействия с сервером

- Создание унифицированного модуля для отправки запросов к серверу
- Автоматическое подключение токена авторизации к каждому запросу
- Автоматическое обновление токена при его истечении
- Обработка ошибок сервера с понятными сообщениями для пользователя

#### 10.2.2. Создание механизма кэширования данных

- Настройка интеллектуального кэширования: данные загружаются один раз и обновляются при изменении
- Автоматическое обновление устаревших данных в фоне
- Индикаторы загрузки во время ожидания данных от сервера

#### 10.2.3. Подключение каждого раздела к серверу

| Раздел интерфейса | Что меняется |
|-------------------|-------------|
| **Боковая панель** | Список проектов загружается с сервера; закрепление проектов сохраняется |
| **Шапка приложения** | Поиск работает через сервер; данные пользователя реальные |
| **Дашборд команды** | Статусы сотрудников обновляются в реальном времени; часы и задачи реальные |
| **Таблица проектов** | Все данные с сервера; фильтры и сортировка на стороне сервера |
| **Детали проекта** | Команда, задачи, файлы, история загружаются с сервера |
| **Виджет бюджета** | Бюджет рассчитывается на стороне сервера |
| **Таблица задач** | Задачи и подзадачи загружаются с сервера; редактирование в таблице сохраняется |
| **Детали задачи** | Чек-лист, комментарии, зависимости, участники — всё с сервера |
| **Управление командой** | Список сотрудников, создание, редактирование — через сервер |
| **Профиль сотрудника** | Статистика рассчитывается на сервере |
| **Клиенты** | Все операции через сервер |
| **Детали клиента** | Контактные лица, проекты, файлы — с сервера |
| **Аналитика** | Все графики и числа формируются из реальных данных сервера |
| **Настройки** | Тема, язык, конфигурация колонок сохраняются на сервере |

#### 10.2.4. Оптимистичные обновления

Для быстрого отклика интерфейса при редактировании прямо в таблице:

- При изменении статуса задачи — интерфейс обновляется мгновенно, запрос к серверу идёт в фоне
- Если сервер отклонил изменение — интерфейс автоматически возвращается к предыдущему значению
- Пользователь не ждёт ответа сервера при каждом действии

#### 10.2.5. Подключение обновлений в реальном времени

- Подписка на события изменения задач, проектов, статусов сотрудников
- Автоматическое обновление данных на экране при получении события
- Уведомления о новых событиях в шапке приложения

#### 10.2.6. Экран авторизации

- Страница входа в систему (email + пароль)
- Страница регистрации организации
- Автоматический вход при наличии действующей сессии
- Перенаправление на экран входа при истечении сессии

### 10.3. Результат этапа

- Все экраны интерфейса работают с реальными данными из базы данных
- Демонстрационные данные полностью удалены
- Несколько пользователей могут работать одновременно, видя изменения друг друга
- Добавлены экраны авторизации

## 11. Этап 7 — Развёртывание и запуск

### 11.1. Что будет сделано

Размещение готового приложения на рабочем сервере, доступном пользователям через интернет.

### 11.2. Состав работ

#### 11.2.1. Серверная инфраструктура

| Компонент | Описание |
|-----------|----------|
| **Сервер приложения** | Виртуальный сервер для запуска серверной части |
| **База данных** | Управляемая база данных PostgreSQL с автоматическим резервным копированием |
| **Файловое хранилище** | Облачное S3-совместимое хранилище для загружаемых файлов |
| **Кэш** | Redis для ускорения часто запрашиваемых данных и управления сессиями |
| **Обратный прокси** | Nginx для распределения нагрузки и защиты от атак |

#### 11.2.2. Безопасность

- Установка SSL-сертификата — все данные передаются по защищённому каналу (HTTPS)
- Настройка файрволла — доступ к серверу только через разрешённые порты
- Защита от распространённых атак (SQL-инъекции, XSS, подбор паролей)
- Ограничение частоты запросов — защита от перегрузки сервера
- Безопасное хранение паролей (хэширование с солью)

#### 11.2.3. Настройка резервного копирования

- Ежедневное автоматическое копирование базы данных
- Хранение резервных копий за последние 30 дней
- Возможность восстановления на любую дату

#### 11.2.4. Мониторинг и оповещения

- Контроль доступности сервера — оповещение при недоступности
- Мониторинг потребления ресурсов (процессор, память, диск)
- Журналирование ошибок для быстрого устранения проблем
- Оповещения команды разработки при критических сбоях

#### 11.2.5. Автоматическое развёртывание

- Настройка автоматической публикации обновлений (CI/CD)
- При внесении изменений в код — автоматическая проверка и развёртывание новой версии
- Возможность быстрого отката к предыдущей версии при обнаружении проблем

#### 11.2.6. Документация

- Инструкция по администрированию сервера
- Описание процедуры восстановления из резервной копии
- Схема архитектуры и взаимосвязей компонентов

### 11.3. Результат этапа

- Приложение доступно пользователям по защищённому адресу в интернете
- Настроено резервное копирование и мониторинг
- Обновления публикуются автоматически
- Передана документация по эксплуатации

## 12. Безопасность и разграничение доступа

### 12.1. Роли пользователей

В системе предусмотрено **три уровня доступа**:

| Роль | Описание | Назначение |
|------|----------|------------|
| **Администратор** | Полный доступ ко всем функциям | Руководитель компании, IT-администратор |
| **Менеджер** | Управление проектами, задачами, клиентами, командой | Руководитель проекта, старший менеджер |
| **Сотрудник** | Работа со своими задачами, учёт времени, просмотр | Архитектор, дизайнер, инженер |

### 12.2. Матрица разграничения прав

| Действие | Администратор | Менеджер | Сотрудник |
|----------|:---:|:---:|:---:|
| Настройки организации и бюджетный лимит | Да | Нет | Нет |
| Создание и удаление пользователей | Да | Нет | Нет |
| Создание и архивирование проектов | Да | Да | Нет |
| Управление участниками проекта | Да | Да | Нет |
| Создание, редактирование, удаление задач | Да | Да | Только свои и назначенные |
| Учёт рабочего времени | Да | Да | Только своё |
| Просмотр аналитики | Да | Да | Ограниченно |
| Управление клиентами | Да | Да | Нет |
| Загрузка файлов | Да | Да | Да |

### 12.3. Принципы безопасности

- Каждая компания (организация) видит только свои данные — полная изоляция
- Все пароли хранятся в зашифрованном виде — даже администратор сервера не может их прочитать
- Сессии автоматически завершаются при длительном бездействии
- Все действия пользователей записываются в журнал для аудита
- Файлы доступны только через временные защищённые ссылки

## 13. Требования к производительности

| Параметр | Требование |
|----------|-----------|
| Время загрузки списка (до 100 записей) | Не более 1 секунды |
| Время сохранения изменения | Не более 0,5 секунды (визуальный отклик мгновенный) |
| Одновременная работа пользователей | Не менее 50 |
| Доступность системы | 99,5% (не более 44 часов простоя в год) |
| Время восстановления из резервной копии | Не более 2 часов |
| Максимальный размер загружаемого файла | 50 МБ |
| Объём базы данных | До 10 ГБ (рост до 50 ГБ без деградации) |
| Объём файлового хранилища | До 100 ГБ (расширяемо) |

## 14. Риски и способы их минимизации

| Риск | Вероятность | Влияние | Способ минимизации |

| Уязвимость в системе безопасности | Низкая | Критическое | Следование стандартам безопасности OWASP, регулярные проверки |
| Превышение планового бюджета на инфраструктуру | Средняя | Среднее | Выбор масштабируемых облачных сервисов, мониторинг расходов |
| Задержки из-за изменения требований | Высокая | Среднее | Поэтапная приёмка, фиксация требований в рамках каждого этапа |

## 15. Критерии приёмки

Каждый этап считается выполненным при соблюдении следующих условий:

### 15.1. Общие критерии (для всех этапов)

- Все описанные операции работают корректно
- Отсутствуют критические ошибки (приводящие к потере данных или невозможности использования)
- Время отклика соответствует указанным требованиям к производительности
- Проведена демонстрация функциональности заказчику

### 15.2. Критерии по этапам

| Этап | Критерий приёмки |
|------|-----------------|
| **1. Инфраструктура** | Можно зарегистрироваться, войти, увидеть свой профиль; тестовые данные в базе |
| **2. Основные операции** | Через API можно выполнить все операции с проектами, задачами, пользователями, клиентами |
| **3. Связи и файлы** | Работают зависимости задач, чек-листы, комментарии; файлы загружаются и скачиваются |
| **4. Учёт времени** | Можно фиксировать время вручную и через таймер; отчёты формируются корректно |
| **5. Реальное время** | Изменения видны другим пользователям без перезагрузки; уведомления доставляются |
| **6. Интеграция** | Все экраны работают с реальными данными; демо-данные удалены; несколько пользователей работают одновременно |
| **7. Развёртывание** | Приложение доступно по HTTPS; резервное копирование настроено; мониторинг работает |
